`db.테이블.where(...)`로 시작하는 모든 호출은 **새 Finder**를 돌려줍니다.
부분 쿼리를 변수에 담아 공유해도 서로 영향을 주지 않습니다.

## 필터

```ts
db.users.where({ active: true })            // 객체 형태
db.users.where("score", ">=", 5)            // 3-인자 형태 (초보자용)
db.users.where("name", "like", "a%")
db.users.where("id", "in", [1, 2, 3])

// 반복 호출은 AND로 합쳐집니다
db.users.where({ active: true }).where("score", ">", 8)
```

3-인자 연산자: `=` `!=` `>` `>=` `<` `<=` `like` `in`

## OR · 중첩 조건

`AND`/`OR`/`NOT` 논리 키로 복잡한 조건을 임의 깊이로 조합해요 (Prisma 스타일).
최상위 컬럼 키와 논리 키는 서로 AND로 묶입니다.

```ts
await db.users.findMany({
  where: {
    active: true,                       // 다른 조건들과 AND
    OR: [
      { score: { gte: 90 } },
      { name: { like: "vip%" } },
    ],
    NOT: { banned: true },
  },
});
```

- `OR: []`(빈 배열)은 아무것도 매칭하지 않아요.
- 더 복잡한 것(윈도우 함수·CTE·서브쿼리)은
  [명명 쿼리](/ko/docs/orm/queries-transactions)나 raw `db.query()`를 쓰세요.

## 마무리 메서드

```ts
await q.all();          // T[]
await q.first();        // T | null
await q.count();        // number
await q.exists();       // boolean
await q.pluck("name");  // 한 컬럼의 값 배열
await q.sum("score");   // 집계: sum / avg / min / max (행 없으면 null)
```

## 페이지네이션

```ts
const page = await db.users.orderBy("id").paginate(2, 20);  // 2페이지, 20개씩
page.rows        // 이 페이지의 행들
page.total       // 전체 개수 (페이지 무관)
page.totalPages  // 총 페이지 수
```

## 정렬·범위·관계

```ts
db.users.orderBy("score", "desc").orderBy("id")   // 다중 정렬
db.users.limit(10).offset(20)
db.posts.where({...}).include({ user: true })      // 관계 로드 (N+1 없음)
db.posts.where({...}).join({ user: true })         // belongsTo를 단일 JOIN으로
```

`include`/`join`의 관계 이름은 생성된 타입 덕분에 에디터에서 자동완성되고,
오타는 컴파일 오류가 됩니다.

## Python에서는

같은 표면이 스네이크 케이스로 존재합니다:

```python
await db.users.where("score", ">=", 5).order_by("score", "desc").limit(10).all()
await db.users.where(active=True).avg("score")
page = await db.users.order_by("id").paginate(1, 20)   # page.rows / page.total
```
