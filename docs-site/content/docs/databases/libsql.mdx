libSQL 백엔드는 원격 sqld 서버(Turso 포함)에 붙습니다. libSQL이 SQLite
방언을 그대로 쓰기 때문에 **SQL 재작성이 전혀 없어요** — `?` 자리표시자,
`BEGIN IMMEDIATE`, 세이브포인트까지 SQLite에서 쓰던 그대로입니다.

## 연결

```
libsql://host[:port]?authToken=…             원격 HTTPS (Turso, sqld)
libsql://host[:port]?tls=false               로컬 sqld — 평문 HTTP
libsql://host?tls=false&authToken=…          조합도 가능
```

Turso라면 대시보드에서 발급한 토큰을 `authToken`에 넣으면 됩니다.

## 동작 방식

- **SQLite 방언 그대로** — 코드·스키마·마이그레이션이 SQLite와 동일.
  `powder ddl`/`migrate`도 SQLite 방언으로 동작해요.
- **연결 시 `PRAGMA foreign_keys=ON`** 을 자동 적용합니다 (SQLite와 달리
  로컬 연결 설정을 직접 만질 수 없으므로 엔진이 대신 켜줘요).
- **동적 타입** — libSQL 값은 SQLite처럼 동적 타입이라, 컬럼의 PCB 타입은
  첫 non-NULL 값에서 추론됩니다.
- libsql 크레이트는 async 전용이지만 엔진이 내부 런타임으로 감싸 다른
  백엔드와 같은 동기 표면을 제공해요 — 사용하는 쪽에선 차이가 없습니다.

## 결과 캐시 없음

원격 서버의 다른 클라이언트 쓰기를 감지할 수 없어 캐시를 끕니다. 반복 쿼리
캐시는 로컬 SQLite 전용 최적화예요.

## 알아둘 것

- TLS 검증 등 보안 스택은 libsql 크레이트의 rustls 체인을 따릅니다. 신뢰할
  수 없는 네트워크의 서버라면 반드시 HTTPS(`tls=false` 미사용) + 토큰을
  쓰세요.
