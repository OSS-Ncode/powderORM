Powderはクエリ結果を行単位でFFI境界を越えて渡すことはありません。Rustコアが
結果全体を**PCBカラムバッファ1つ**にエンコードして1回だけ転送し、各言語の
リーダーがそのバッファ上にzero-copyのビューを作ります。

## 実測値（sort、5万行 × 5回、中央値ms）

| ランタイム | Powder | raw driver | 代表的なORM |
|---|---|---|---|
| Rust | **2.84** | rusqlite 3.94 | Diesel 8.75 |
| TS/JS | **2.91** | better-sqlite3 11.20 | Prisma 24.57 |
| Java | **3.98** | JDBC 17.20 | ORMLite 25.07 |
| Go | **4.08** | database/sql 29.46 | GORM 50.56 |
| Python | **4.23** | sqlite3 23.07 | Peewee 67.90 |

5言語すべてのタブで**その言語のraw SQL driverより高速です。** すべての
ランナーは同じデータ・同じSQLで実行され、行数+チェックサムのクロス
検証をパスする必要があります。

同一の読み取りクエリを繰り返す場合（SQLite `:memory:`）、結果キャッシュ
から**0.02〜0.03ms**で応答します — 書き込み時の全体無効化、ファイルDBの
`data_version`監視、非決定的な関数のブロックという三重のゲートで正確性が
保証されます。

## ORMレイヤーのコストは？

ほぼありません。SQLスケルトンは`powder generate`の時点で生成され
（AOT）、ランタイムは3種類のキャッシュによって反復クエリの文字列処理を
排除します。

- **行ファクトリキャッシュ** — テーブルごとに1回コード生成される行生成
  関数（固定hidden class）
- **where-shapeキャッシュ** — WHERE断片を述語の形状としてメモ化（値は
  キーに含めない）
- **ORDER BYキャッシュ** — ソート断片も同様にメモ化

## 自分で動かしてみる

```bash
cd bench-site && npm start    # http://localhost:4173
```

TS/JS · Python · Rust · Java · Goのタブ、4つのシナリオ（ソート/
フィルタ/集計/結合）、棒グラフ・グループ・テーブル・ラインスイープ
ビューを提供します。
